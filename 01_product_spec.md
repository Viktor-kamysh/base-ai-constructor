# Construction Pro Agent v1 — Product Specification

## 1. Цель
Создать AI-помощника для строителей, инженеров, технадзора и девелоперов, который ускоряет:
- ведение stavební deník;
- подготовку отчетов контрольного дня;
- первичную сверку смет подрядчиков;
- создание черновых сметных структур;
- сохранение проектового контекста и продолжение работы по проекту.

## 2. Неправильное позиционирование
Агент НЕ:
- заменяет сметчика, инженера, технадзор или PM;
- выдает финально утвержденные документы без проверки человеком;
- придумывает цены, объемы, листы документации или факты.

## 3. Правильное позиционирование
Construction Pro Agent — это **AI co-pilot** для:
- document-assisted estimate review;
- structured site diary creation;
- contractor quote audit;
- project memory and follow-up.

## 4. Разрешенные источники данных
### 4.1 Источники v1
- загруженные пользователем PDF, XLSX, DOCX, TXT, JPG, PNG;
- голосовые заметки и их транскрипты;
- фото с объекта;
- внутренние шаблоны приложения;
- вручную загруженные reference price tables.

### 4.2 Источники v2+
- лицензированная интеграция с ÚRS;
- исторические данные прошлых проектов пользователя.

### 4.3 Запреты
Агенту запрещено использовать знания модели как доказательство стоимости, объема, состава работ или факта выполнения.

## 5. Основные модули v1
1. **Project Workspace**
2. **Site Diary Assistant**
3. **Estimate Review Assistant**
4. **Estimate Draft Assistant**
5. **Template & Export Engine**
6. **Issue Log**
7. **Version History**

## 6. Обязательные статусы ответа
Каждый вывод агента должен иметь один из статусов:
- `found`
- `missing`
- `unclear`
- `duplicate`
- `needs_review`
- `insufficient_data`

## 7. Confidence scale
- `high`
- `medium`
- `low`

Правило: если нет подтверждающих данных, confidence не может быть `high`.

## 8. Что агент делает при нехватке данных
Вместо догадки агент обязан:
1. назвать, каких данных не хватает;
2. сказать, для какой проверки они нужны;
3. указать, из какого документа/источника их обычно берут;
4. остановить окончательный вывод.

## 9. Ключевые user stories
### 9.1 Site Diary
Как инженер, я хочу надиктовать заметки и загрузить фото, чтобы получить готовый stavební deník и summary для рассылки.

### 9.2 Estimate Review
Как девелопер, я хочу загрузить смету подрядчика, чтобы агент выделил дубли, unclear items, возможные пропуски и подготовил вопросы подрядчику.

### 9.3 Draft Estimate
Как специалист, я хочу создать черновую структуру сметы по данным проекта, чтобы быстрее стартовать ручную доработку.

### 9.4 Project Continuity
Как пользователь, я хочу открыть проект через несколько дней и продолжить работу без потери контекста.

## 10. Выходные документы v1
- PDF
- DOCX
- email-ready text
- structured JSON record

## 11. Branding документов
Пользователь может загрузить:
- логотип;
- изображение подписи;
- изображение печати.

На v1 это визуальная вставка в документ, не квалифицированная электронная подпись.

## 12. Метрики успеха v1
- сокращение времени на подготовку stavební deník;
- сокращение времени на первичную сверку смет;
- доля документов, требующих только легкой правки перед отправкой;
- число найденных замечаний до ручной проверки;
- возврат пользователя в проектовую папку.
